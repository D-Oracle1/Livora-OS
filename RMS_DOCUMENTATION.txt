1. EXECUTIVE SUMMARY
================================================================================

RMS (Realtors Management System) is an enterprise-grade, multi-tenant PropTech
platform designed for real estate companies in Nigeria. It provides end-to-end
management for realtors, clients, properties, sales, commissions, HR, and more.

Key Statistics:
  - 23+ backend modules
  - 165+ API endpoints
  - 80+ frontend pages across 7 route groups
  - 5 user roles (SUPER_ADMIN, ADMIN, REALTOR, CLIENT, STAFF)
  - Multi-tenant architecture with isolated databases per company
  - Real-time features via Pusher WebSockets
  - PWA-enabled with offline support
  - Deployed on Vercel (frontend + backend serverless)
  - Event management with QR code check-in system
  - Registration confirmation emails with embedded QR codes

================================================================================
2. ARCHITECTURE OVERVIEW
================================================================================

Multi-Tenant Architecture:
  - Master Database: Stores company/tenant records and super admin accounts
  - Tenant Database: Each company gets an isolated PostgreSQL database
  - MongoDB: Used for real-time chat messages
  - Redis: Caching layer and job queue (BullMQ)

Request Flow:
  1. TenantMiddleware -> Resolve company from domain/invite code
  2. Global Pipes -> Validate DTOs, transform data
  3. Route Guards -> JWT auth, role check, permission check
  4. Route Handler -> Business logic
  5. Interceptors -> Logging, audit, response transformation
  6. Global Filter -> Error handling
  7. Response -> Standardized JSON: { success, data, meta?, timestamp }

NOTE: TransformInterceptor flattens paginated responses. If a service returns
{ data: [...], meta: {...} }, the interceptor lifts these to top-level:
{ success: true, data: [...items], meta: { totalPages, ... }, timestamp }
Frontend must read res.data and res.meta (NOT res.data.data / res.data.meta).

Monorepo Structure:
  /backend    - NestJS API (TypeScript)
  /frontend   - Next.js 16 App Router (React 19, TypeScript)
  /shared     - Shared types/configs
  /docker     - Docker Compose files (dev & prod)

  POST /pusher/auth            - Authorize Pusher channel subscription
  POST /auth/2fa/setup         - Generate TOTP secret + QR
  POST /auth/2fa/enable        - Enable 2FA (verify first TOTP)
  POST /auth/2fa/disable       - Disable 2FA
  POST /auth/2fa/verify        - Verify TOTP code at login
  POST /auth/2fa/recovery-code - Use backup recovery code
  POST /auth/2fa/regenerate-backup-codes - New backup codes
  GET  /auth/oauth/google      - Google OAuth redirect
  GET  /auth/oauth/google/callback - Google OAuth callback
  GET  /auth/oauth/facebook    - Facebook OAuth redirect
  GET  /auth/oauth/facebook/callback - Facebook OAuth callback

Features:
  - Dual authentication (SUPER_ADMIN from master DB, users from tenant DB)
  - Password strength validation (8+ chars, upper, lower, number, special)
  - bcrypt hashing (12 rounds)
  - JWT access tokens (7 day expiry)
  - UUID refresh tokens (30 day expiry, stored in DB)
  - SHA256 hashed password reset tokens (1 hour expiry)
  - Invite code registration (first user becomes ADMIN)
  - Role-based profile creation on register
  - TOTP-based 2FA (RFC 6238, built-in implementation, no external lib)
  - 8 backup recovery codes per user
  - Google + Facebook OAuth (passport-google-oauth20, passport-facebook)

5.28 EVENT MODULE (src/modules/event/)
----------------------------------------------------------------------
Status: FULLY IMPLEMENTED (added 2026-03-17)

Services:
  EventService        - CRUD for events
  RegistrationService - Public sign-up + QR check-in
  QrService           - JWT-signed QR token generation and verification
  EventAnalyticsService - View tracking, metrics (cached 120s)

Public Endpoints (no auth):
  GET  /events/homepage                    - Featured/upcoming events for landing page
  GET  /events/public                      - Published events list
  GET  /events/public/:slug                - Event detail by slug (increments view count)
  POST /events/public/:eventId/register    - Register for an event
    Rate limit: 5 req/60s
    Body: { userData: Record<string, unknown> }
    Returns: { registrationCode, status, qrCode (base64 PNG), createdAt }

Admin Endpoints (ADMIN, HR, STAFF, GENERAL_OVERSEER):
  POST  /events                                     - Create event
  GET   /events                                     - List events (paginated, filtered)
  GET   /events/:id                                 - Event details
  PUT   /events/:id                                 - Update event
  DELETE /events/:id                                - Delete event + all data
  PATCH /events/:id/status                          - Change status (draft/published/closed)
  GET   /events/:id/analytics                       - Event analytics (views, regs, check-ins)
  GET   /events/:id/registrations                   - List registrations (paginated, status filter)
  PATCH /events/registrations/:id/status            - Approve/reject a registration
  POST  /events/checkin                             - Check-in by QR token OR registration code
    Rate limit: 120 req/60s
    Accepts: JWT QR token from camera scan OR EVT-... registration code (manual entry)
    Returns: { alreadyCheckedIn, checkedInAt, registrationCode, eventTitle, message }

QR Code System:
  - Each registration gets a unique JWT-signed token (10-year expiry)
  - QR code is a 300x300px PNG rendered as a base64 data URL
  - Check-in accepts both camera-scanned QR token and typed registration code
  - DB lookup is the authoritative validation (JWT signature not re-verified at check-in)
  - Registration codes format: EVT-{timestamp}-{8-char-hex} (e.g. EVT-1747123456789-A1B2C3D4)

Registration Email:
  - Confirmation email sent fire-and-forget immediately after registration
  - Contains: event title, date, format, location, registration code, form data
  - QR code embedded as inline CID attachment (works in Gmail, Outlook, Apple Mail)
  - Uses registrant's email from any form field with fieldType === 'email'

Form Builder:
  - Admin defines custom form fields per event (text, email, phone, number,
    textarea, select, checkbox, date)
  - Required/optional toggle per field
  - Order index for drag-drop reordering
  - Validation: email format, phone format, required fields
  - Deduplication: prevents re-registration by email per event

Analytics (EventAnalytics):
  - viewsCount: incremented on each public event detail page load
  - registrationsCount: count of non-rejected registrations
  - checkinsCount: count of checked-in registrations
  - conversionRate: registrations / views * 100
  - Cached for 120 seconds, recomputed on each registration/check-in

5.30 COMMON INFRASTRUCTURE

  @Roles(...)       - Require specific roles
  @CurrentUser()    - Inject authenticated user
